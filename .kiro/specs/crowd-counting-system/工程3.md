## 难点 3 总结：视频帧与检测结果的时间同步（选型：先 D，效果不好再 考虑C，G）

> 现实现说明：当前采用服务端渲染后推流（方案 F/G 组合），时间同步由渲染链路内完成，前端不再做对齐渲染。本文为历史分析保留。

### 1) 背景、目标与整体数据链路

#### 背景与目标

前端本地连续播放摄像头画面（30fps），后端对抽样帧（1–3fps）做 YOLO 推理，并通过网络离散推送检测框/热力数据；前端把结果叠加到原始画面上播放。
你的目标是：**框/热力与对应画面准确对齐**，并且在公网延迟抖动下仍尽量稳定、观感连贯。

#### 为什么会“飘”

视频播放是连续时间轴，而检测结果是离散事件且延迟不稳定；如果“收到结果就画在当前帧”，结果属于过去某帧却被画到当前帧上，延迟抖动会让偏差忽大忽小，于是出现“飘”和跳动。

#### 整体数据链路（面向 D/C）

1. 抽帧时记录对齐键：`seq + capture_video_ts(=video.currentTime)`（可再加 `capture_wall_ts=performance.now()`用于测延迟）
2. 后端推理返回结果时带回同一对齐键
3. 前端把结果写入短缓存 `result_buffer`
4. 每次 `requestAnimationFrame` 渲染时按 `video.currentTime`（或 `video.currentTime - D`）从缓存取最匹配结果来画
5. 方案 D：对结果做 EMA/插值让 1–3fps 看起来更顺滑；若迟到明显再上方案 C 加固定缓冲 D

### 2) 你的理解变化（每点一句话）

- 你明确了“视频连续、结果离散”是根因，收到就画必然漂移。
- 你意识到公网延迟抖动会让偏移不稳定，因此必须引入对齐机制。
- 你接受用时间戳/帧号把结果绑定到对应帧，再在前端按视频时间轴匹配绘制。
- 你进一步认识到 1–3fps 会导致视觉跳变，因此优先尝试平滑/插值（方案 D）。
- 你也理解了备用策略：加 200–500ms 播放缓冲（方案 C）可显著提升抗抖动稳定性。
- 你注意到后端只存“最新结果”可能被覆盖，因此前端需要短缓存（或后端保短历史）来完成对齐。

### 4) 沉淀核心结论（结合你的选型）

1. **对齐是底座**：先把结果绑定到 `seq/capture_video_ts`，前端按视频时间轴取匹配结果再画，漂移问题才有根治路径。
2. **你选 D 是正确的第一步**：在对齐基础上做 EMA/插值，用最小带宽获得更顺滑观感，适配 1–3fps 推理。
3. **如果结果“经常迟到/忽隐忽现”**：再启用 C，加 200–500ms 播放缓冲，用少量延迟换明显稳定性。
4. **缓存策略要配套**：后端只存最新结果可以，但前端要保留短结果缓存用于对齐；否则会被覆盖导致无法回看“稍早的那条结果”。
