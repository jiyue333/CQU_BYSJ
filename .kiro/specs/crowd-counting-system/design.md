# 设计文档

## 概述

当前实现采用**方案 F：服务端渲染热力图**。后端在媒体网关内生成原始流，并由渲染服务拉流做 YOLO 推理 + 热力图叠加，再把渲染后的流推回网关，前端直接播放渲染流。

核心约束：
- 播放优先低延迟（WebRTC），HTTP-FLV/HLS 作为兜底。
- 热力图由服务端渲染，前端不再做 Canvas 叠加。
- 渲染输出流命名为 `{stream_id}_heatmap`。

## 数据链路（现实现）

1) 前端创建/启动流，后端调用 ZLMediaKit 创建原始流。
2) 后端把 `play_url` / `webrtc_url` 指向渲染流 `{stream_id}_heatmap`。
3) 后端发布 `render:control` START 指令（Redis PubSub）。
4) RenderWorker 从 ZLM 拉 RTSP（原始流），推理后把渲染帧推 RTMP 回 ZLM。
5) RenderWorker 写入 `result:{stream_id}` 与 `latest_result:{stream_id}`，并复用 `inference:status` 上报状态。
6) 后端 WebSocket 推送结果与状态给前端；前端仅展示总人数和连接状态。

## 关键约定

- **渲染流命名**: `{stream_id}_heatmap`
- **外部播放地址**: `ZLM_EXTERNAL_URL` 生成 HLS/WebRTC URL
- **容器内拉/推流**: `ZLM_BASE_URL` 推导 host
- **vhost 处理**: RTSP/RTMP 内部 URL 追加 `vhost=__defaultVhost__` 以避免 "no such stream"

## 当前 UI 能力

- 已实现：流管理、播放（WebRTC/FLV/HLS）、实时总人数、配置面板。
- 未接入：ROI 配置/展示、历史图表与导出、热力图开关等高级交互。
