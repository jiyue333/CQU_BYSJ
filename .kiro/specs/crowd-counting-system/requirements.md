# 需求文档

## 简介

基于 YOLO 的实时人流计数与密度分析系统。采用**播放帧率与推理频率解耦**的设计：视频在前端流畅播放，后端每秒抽取 1-3 帧进行推理，热力图按秒刷新并平滑叠加在原始画面上。系统部署在公网，支持多路视频输入。

**核心架构**：所有视频源（本地文件、浏览器摄像头、RTSP 摄像头）统一抽象为 `stream_id`，前端只认 `play_url`，推理只认 `getSnap(stream_id)`。

## 术语表

- **系统 (System)**: 人流计数与密度分析系统
- **用户 (User)**: 系统操作人员
- **视频源 (Video_Source)**: 视频输入源，包括本地视频文件、浏览器摄像头、RTSP 摄像头
- **stream_id**: 视频流的唯一标识，所有视频源统一抽象为 stream_id
- **play_url**: 前端播放地址，由媒体网关生成
- **检测框 (Detection_Box)**: 标识检测到的人体位置的边界框
- **感兴趣区域 (ROI)**: Region of Interest，用户定义的监控区域
- **密度 (Density)**: 区域内人数与区域面积的比值
- **热力图 (Heatmap)**: 可视化人群分布密度的图层
- **媒体网关 (Media_Gateway)**: 负责视频流接入、转码、分发的开源组件
- **推理频率**: 后端 YOLO 推理的频率，1-3fps
- **播放帧率**: 前端视频播放的帧率

## 需求列表

### 需求 1: 视频输入与播放

**用户故事:** 作为用户，我希望能够输入多种类型的视频源并流畅播放，以便监控不同场景。

#### 验收标准

1. 当用户上传本地视频文件时，系统应将其转为可播放流并在前端播放
2. 当用户授权浏览器摄像头时，系统应将其发布到媒体网关并在前端播放
3. 当用户输入 RTSP 摄像头地址时，系统应通过媒体网关拉流并在前端播放
4. 系统应为每个视频源生成统一的 stream_id 和 play_url
5. 用户不需要感知媒体网关的存在，只需操作"开始/停止"按钮
6. 如果视频源不可用，则系统应显示错误信息并允许重试

### 需求 2: 人体检测与计数

**用户故事:** 作为用户，我希望系统能够检测视频中的人并统计数量，以便了解当前人数。

#### 验收标准

1. 当视频流进入 running 状态时，推理服务应开始按频率调用 getSnap(stream_id) 抓帧
2. 系统应每秒抽取 1-3 帧进行 YOLO 推理
3. 系统应实时显示检测到的人数总计
4. 当检测置信度低于阈值时，系统应过滤掉低置信度的检测结果

### 需求 3: 区域划分与密度分析

**用户故事:** 作为用户，我希望能够定义监控区域，以便分析特定区域的密度。

#### 验收标准

1. 当用户在视频画布上绘制感兴趣区域时，系统应保存区域配置
2. 系统应支持多个感兴趣区域定义（如前区/中区/后区）
3. 系统应计算并显示每个感兴趣区域的密度
4. 系统应根据可配置的阈值对密度等级进行分类（低/中/高）
5. 当感兴趣区域配置变更时，系统应立即应用新配置

### 需求 4: 热力图可视化

**用户故事:** 作为用户，我希望看到人群分布的热力图叠加在视频上，以便识别高流量区域。

#### 验收标准

1. 系统应根据检测位置生成热力图数据
2. 系统应将热力图叠加在前端视频播放画面上
3. 热力图应每秒更新一次，并平滑过渡
4. 系统应提供显示/隐藏热力图的开关
5. 热力图叠加不应影响视频播放的流畅性（30fps）

### 需求 5: 实时数据展示

**用户故事:** 作为用户，我希望看到实时统计数据，以便监控当前情况。

#### 验收标准

1. 系统应显示当前总人数
2. 系统应显示每个已定义感兴趣区域的密度值
3. 统计数据应每秒更新 1-3 次

### 需求 6: 历史数据查询与导出

**用户故事:** 作为用户，我希望能够查询和导出历史数据，以便分析趋势。

#### 验收标准

1. 系统应存储带有时间戳的检测统计数据
2. 当用户指定时间范围时，系统应显示该时段的历史统计数据
3. 系统应提供图表（折线图/柱状图）来可视化历史趋势
4. 当用户请求数据导出时，系统应生成可下载的文件（CSV/Excel）

### 需求 7: 多路视频支持

**用户故事:** 作为用户，我希望系统能够支持多路视频同时处理。

#### 验收标准

1. 系统应支持同时处理多路视频流
2. 系统应在公网环境下稳定运行

### 需求 8: 配置管理

**用户故事:** 作为用户，我希望能够配置系统参数，以便使系统适应不同场景。

#### 验收标准

1. 系统应允许配置检测置信度阈值
2. 系统应允许配置密度等级阈值
3. 系统应允许配置推理频率（1-3fps 范围内）
4. 系统应在会话之间持久化配置

### 需求 9: 错误处理与稳定性

**用户故事:** 作为用户，我希望系统能够优雅地处理错误，以便监控不被中断。

#### 验收标准

1. 如果视频流断开，则系统应尝试自动重连
2. 如果多次重连失败，则系统应通知用户
3. 当发生处理错误时，系统应记录错误并继续运行
4. 系统应提供连接和处理状态的指示器


### 需求 10: 多用户支持（下一迭代）

**用户故事:** 作为用户，我希望系统能够支持多用户同时访问和查看。

#### 验收标准

1. 系统应支持多个用户同时访问和查看
2. 每个用户应能独立配置自己关注的视频源和区域

> **备注**: 此需求计划在下一个迭代实现，当前迭代优先完成单用户多路视频的核心功能。
