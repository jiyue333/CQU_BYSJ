# 基于 YOLO 的实时人流计数与密度分析系统

## 核心设计策略

**播放帧率与推理频率解耦**：
- 视频播放：前端本地连续播放（30fps），保证流畅观看体验
- 推理频率：后端每秒抽取 1-3 帧进行 YOLO 推理，秒级精度足够
- 热力图更新：按秒刷新，平滑过渡叠加在原始画面上

**优势**：既能"看起来是视频"，又能把推理/带宽/并发成本控制在可扩展范围内。

## 三层交互说明

| 交互 | 方向 | 频率 | 内容 |
|------|------|------|------|
| 视频播放 | 视频源 → 前端 | 30fps | 原始视频流直接播放 |
| 推理结果 | 后端 → 前端 | 1-3次/秒 | 人数、密度、热力图数据 |
| 区域配置 | 前端 → 后端 | 按需 | ROI 配置信息 |
| 历史查询 | 前端 ↔ 后端 | 按需 | 历史统计数据 |

## 功能清单

### 一、前端

1. 视频播放（30fps 本地播放，不经过后端）
2. 热力图叠加层（每秒更新，平滑过渡）
3. 实时数据面板（人数、进出、密度）
4. 区域划分配置界面
5. 历史数据查询与导出

### 二、后端（公网部署，支持多路多用户）

1. 多路视频流接入（RTSP/HLS/文件）
2. 抽帧服务（1-3fps）
3. YOLO 推理 + 追踪
4. 人流计数与区域密度计算
5. 热力图数据生成
6. 数据存储
7. WebSocket 推送（1-3次/秒）+ REST API

### 三、模型层

1. YOLO 模型选型与微调
2. 目标追踪集成
3. 数据集构建

## 工程难点（深度分析版）

1. **视频流传输与处理**
   - 浏览器摄像头：逐帧 Base64 带宽大（1080p 约 2-5MB/帧），MediaRecorder 分片有 1-3 秒延迟，WebRTC 需要信令服务器 + STUN/TURN
   - 帧队列回压：解码速度高于推理速度时队列会堆积，延迟持续增长，需要设计丢帧/降采样/动态 FPS 策略
   - 编码传输格式：WebSocket 二进制 + JPEG/H264 更省带宽但编码耗 CPU，Base64 简单但带宽开销大

2. **Python 异步与阻塞推理的融合**
   - OpenCV `cv2.read()` 和 YOLO `model.predict()` 都是同步阻塞操作
   - FastAPI WebSocket 长连接容易被阻塞，需要 `run_in_executor` 或多进程池
   - 进程间队列/共享内存增加实现与调试复杂度
   - 可选方案：vidgear 的 WebGear 封装了异步视频流处理

3. **视频帧与检测结果的时间同步**
   - 前端播放视频是连续的，检测结果是离散推送的
   - 网络延迟不稳定，检测框会"飘"（对不上当前帧）
   - 需要时间戳或帧号对齐机制
   - 或者改为后端渲染检测框后推送 MJPEG 流（简单但带宽大）

4. **RTSP 流的稳定性处理**
   - OpenCV `VideoCapture` 在网络抖动时 `read()` 会阻塞整个线程
   - 需要单独线程 + 超时机制 + 断线重连逻辑
   - 可选方案：FFmpeg subprocess 或 imageio-ffmpeg，但接口不如 OpenCV 直观

## 模型难点

1. **模型选型与结构能力**
   - 轻量模型（YOLOv5s / YOLOv8n / YOLO-NAS）在速度-精度-部署复杂度之间取舍
   - 密集/遮挡场景下的小目标检测能力不足（漏检、框黏连）

2. **数据集与标注体系**
   - 公开数据集与真实监控/公网场景存在域差异（角度、分辨率、光照、码率）
   - 数据集构建与标注成本高，且容易出现标注口径不一致（算不算人、遮挡算不算）

3. **训练与鲁棒性优化**
   - 微调是否有效、怎么选策略：迁移学习/增强/采样等对密集场景影响大
   - 跨场景泛化难：新场景一换性能掉点，需要域适配或持续迭代

4. **推理性能与部署加速**
   - 输入分辨率、batch、抽帧频率带来的吞吐与成本权衡（多路并发时更明显）
   - ONNX/TensorRT/FP16/INT8 等优化带来的精度损失与工程稳定性取舍
