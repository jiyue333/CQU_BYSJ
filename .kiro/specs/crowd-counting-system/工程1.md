# 人流计数系统工程设计文档

## 1) 背景、目标、原理与整体数据链路

### 你要做的事

做一个**基于 YOLO 的人流计数与密度分析系统**：从用户的摄像头/视频中获得画面，在后端做检测与统计，输出**人数、密度/热力图**，并把热力图叠加到原始画面上"播放"。

### 核心原理（为什么"视频"最后会变成"帧"）

- YOLO 的输入本质是**单帧图像**（最终变成 tensor：NCHW、RGB、resize/归一化等）。
- "视频推理"只是工程上的说法：把连续视频**按策略抽帧**喂给模型，然后把结果按时间聚合成**计数/密度**。

### 整体数据链路（实际实现）

以当前目标（每秒 1-3 帧推理、秒级精度）为主线，链路可以拆成 6 段：

**1. 数据源（用户本地/监控摄像头）**

- RTSP 摄像头：通过 RTSP 协议推流
- 本地文件：上传到服务器后由 FFmpeg 推流
- 浏览器摄像头：通过 WebRTC WHIP 协议推流到 ZLMediaKit

**2. 媒体网关（ZLMediaKit）**

- 接收各类视频源，统一转换为多协议输出（HLS/HTTP-FLV/WebRTC/RTSP）
- 提供 getSnap API 用于服务端抽帧（JPEG 快照）
- 充当 SFU 角色，支持多人观看同一路流

**3. 服务端抽帧（Inference Worker）**

- 按 1-3fps 频率调用 ZLMediaKit getSnap API 获取 JPEG 快照
- 背压策略：只保最新帧，忙就丢旧帧
- 支持多路视频并发推理

**4. 解码与预处理（服务端）**

- JPEG bytes 通过 OpenCV/PIL 解码成像素矩阵（BGR）
- YOLO 内部自动处理 resize/letterbox 转换为 tensor

**5. YOLO 推理与统计（服务端）**

- YOLOv8n 输出检测框和置信度
- 聚合成每秒计数、密度网格（heatmap grid）
- EMA 平滑/滤波

**6. 结果回传与叠加播放（客户端）**

- 前端通过 WebRTC/HTTP-FLV/HLS 播放原始视频（25-30fps）
- 通过 WebSocket 接收推理结果（1-3fps）
- Canvas 叠加热力图，EMA 平滑过渡
- **关键：播放帧率与推理刷新频率解耦**

## 2) 核心设计决策

### 为什么选择"服务端抽帧"而非"客户端抽帧"

| 方案 | 优点 | 缺点 |
|------|------|------|
| 客户端抽帧上传 | 减少服务端压力 | 前端复杂度高、需要处理各种浏览器兼容性 |
| 服务端抽帧（当前实现） | 实现简单、统一处理各类视频源 | 依赖 ZLMediaKit getSnap API |

**选择服务端抽帧的原因：**

1. ZLMediaKit 已经接收了视频流，直接调用 getSnap 最简单
2. 统一处理 RTSP/文件/Webcam 三种视频源
3. 前端只需关注播放和热力图叠加，职责清晰

### 为什么需要 ZLMediaKit（媒体网关）

1. **协议转换**：监控摄像头通常只支持 RTSP，浏览器无法直接播放
2. **多协议输出**：同一路流可以同时提供 WebRTC（低延迟）、HTTP-FLV、HLS（兼容性好）
3. **抽帧能力**：getSnap API 提供 JPEG 快照，供推理使用
4. **SFU 能力**：支持多人同时观看同一路流

## 3) 方案对比

公网 + 多路并发 + 1-3fps 推理 + 热力图叠加播放场景下的方案对比：

| 方案 | 传输内容 | 热力图叠加 | 部署复杂度 | 并发扩展性 | 适配度 |
|------|----------|-----------|-----------|-----------|--------|
| A. 抽帧图片上传 | 每秒 1-3 张 JPEG | 满足 | 低 | 高 | 仅 Webcam |
| B. 短片段上传 | 1-2秒 WebM/MP4 | 满足 | 中 | 中-高 | 可选 |
| C. WebRTC 直连 | 连续码流 | 满足 | 高 | 中 | 能力过剩 |
| D. WebRTC+SFU | 浏览器推SFU | 满足 | 高 | 高 | 生产常见 |
| **E. 混合方案** | ZLM多协议+getSnap | Canvas叠加 | 中 | 高 | **推荐（低延迟）** |
| **F. 服务端渲染** | ZLM+服务端叠加推流 | 服务端渲染 | 中-高 | 中 | **推荐（简单前端）** |

### 方案 E vs 方案 F 选择指南

| 场景 | 推荐方案 |
|------|----------|
| 需要低延迟（<1s） | 方案 E |
| 前端需要极简实现 | 方案 F |
| 多路并发（>4路） | 方案 E |
| 需要热力图与视频完美对齐 | 方案 F |
| 服务端资源有限 | 方案 E |
| 需要支持非浏览器客户端 | 方案 F |

## 4) 方案 F：服务端渲染热力图（无前端叠加）

### 适用场景

- 可接受 1-5 秒视频延迟
- 可接受热点推理较慢（24fps 视频中只有 8fps 渲染热点密度）
- 希望前端实现极简（只播放视频，无需 Canvas 叠加逻辑）
- 需要热力图与视频帧完美对齐

### 核心思路

服务端直接在视频帧上绘制热力图，然后重新编码推流到 ZLMediaKit，前端只需播放处理后的视频流。

### 数据流架构图

```
视频源 (RTSP/File/Webcam)
            |
            | RTSP/WHIP/FFmpeg
            v
    +---------------+
    |  ZLMediaKit   |
    |  (原始流)     |
    +-------+-------+
            |
            | RTSP/RTMP 拉流
            v
    +-------------------+
    |  渲染 Worker      |
    |  (24fps 解码)     |
    +-------------------+
            |
      +-----+-----+
      |           |
      v           v
  每3帧抽1帧    其余帧
  (8fps推理)   (复用热力图)
      |           |
      v           v
  YOLO推理    热力图缓存
      |           |
      +-----+-----+
            |
            v
    OpenCV 叠加热力图
            |
            v
    FFmpeg 编码推流
            |
            v
    +-------------------+
    |  ZLMediaKit       |
    |  (渲染后流)       |
    |  stream_id_heatmap|
    +-------------------+
            |
            v
    前端播放 (带热力图)
    (无需 Canvas 叠加)
```

### 方案 E vs 方案 F 对比

| 维度 | 方案 E (Canvas叠加) | 方案 F (服务端渲染) |
|------|---------------------|---------------------|
| 前端复杂度 | 高（Canvas叠加+EMA平滑） | 低（只播放视频） |
| 服务端负载 | 低（只推理1-3fps） | 高（推理8fps+编码24fps） |
| 延迟 | 低（<1s） | 中（1-5s） |
| 热力图对齐 | 需要时间同步 | 完美对齐（同帧处理） |
| 带宽 | 1x | 2x（原始流+渲染流） |
| 兼容性 | 需要现代浏览器 | 任何视频播放器 |
| 多路并发 | 高（前端分担渲染） | 中（服务端编码瓶颈） |

### 技术实现要点

**1. 渲染 Worker 架构**

```python
# 伪代码：渲染 Worker 核心逻辑
class RenderWorker:
    def __init__(self, stream_id: str):
        self.stream_id = stream_id
        self.heatmap_cache = None  # 缓存最新热力图
        self.frame_count = 0
        
    async def run(self):
        # 拉取原始流（24fps）
        cap = cv2.VideoCapture(rtsp_url)
        # 推流到 ZLMediaKit
        ffmpeg_proc = start_ffmpeg_push(f"{stream_id}_heatmap")
        
        while True:
            ret, frame = cap.read()
            self.frame_count += 1
            
            # 每3帧做一次推理（8fps）
            if self.frame_count % 3 == 0:
                detections = await self.yolo_inference(frame)
                self.heatmap_cache = self.generate_heatmap(detections)
            
            # 所有帧都叠加热力图
            if self.heatmap_cache is not None:
                frame = self.overlay_heatmap(frame, self.heatmap_cache)
            
            # 编码推流
            ffmpeg_proc.stdin.write(frame.tobytes())
```

**2. 热力图叠加（复用现有逻辑）**

现有 `backend/scripts/demo_inference.py` 中的 `draw_heatmap_overlay` 可直接复用：

```python
def overlay_heatmap(frame: np.ndarray, heatmap_grid: list[list[float]]) -> np.ndarray:
    # 将 0-1 网格转换为颜色图
    heatmap = np.array(heatmap_grid)
    heatmap_colored = cv2.applyColorMap(
        (heatmap * 255).astype(np.uint8), 
        cv2.COLORMAP_JET
    )
    # 叠加到原始帧
    return cv2.addWeighted(frame, 0.7, heatmap_colored, 0.3, 0)
```

**3. FFmpeg 编码推流**

```bash
ffmpeg -f rawvideo -pix_fmt bgr24 -s 1920x1080 -r 24 -i pipe:0 \
       -c:v libx264 -preset ultrafast -tune zerolatency \
       -f flv rtmp://127.0.0.1/live/{stream_id}_heatmap
```

**4. 流管理扩展**

```python
# StreamService 扩展
class StreamService:
    async def start_render_stream(self, stream_id: str) -> str:
        """启动渲染流，返回渲染后的播放地址"""
        render_worker = RenderWorker(stream_id)
        await render_worker.start()
        return f"/live/{stream_id}_heatmap.flv"
```

### 潜在风险与注意事项

1. **性能瓶颈**：24fps 解码 + 叠加 + 编码在 CPU 上成本高，多路并发时可能需要 GPU 编码（NVENC/VAAPI）

2. **延迟控制**：
   - 编码延迟：使用 `-preset ultrafast -tune zerolatency`
   - HLS 切片延迟：`zlm-config/config.ini` 中 `hls.segDur=2` 会增加 2-4 秒延迟
   - 推荐使用 HTTP-FLV 或 WebRTC 降低延迟

3. **资源回收**：渲染进程/子进程的生命周期和异常清理必须可靠，否则易有推流残留或 CPU 泄漏

4. **同步策略选择**：
   - 严格对齐：渲染链路内每 3 帧做一次推理（8fps）
   - 允许漂移：渲染链路消费 Redis 的 `result:{stream_id}`，热力图可能滞后 100-300ms

### 推荐实现路径

1. **最小可行原型**：用 OpenCV + FFmpeg pipe 拉取 RTSP → 叠加 → 推 RTMP 到 ZLM，验证 24fps 输出与延迟

2. **决定同步策略**：若对齐要求严格，渲染链路内每 3 帧做一次推理；若可接受漂移，渲染链路只消费 Redis

3. **接入业务**：在 `StreamService` 增加 `enable_render` 或独立"渲染流"概念，返回渲染后 `play_url`

4. **前端适配**：默认播放渲染流并关闭 `HeatmapOverlay`（或提供开关切换方案 E/F）

5. **逐步优化**：加入编码参数配置、并发限制、监控与告警，必要时引入 GPU 编码

## 5) 沉淀核心结论

1. **任务定位**：不是"实时视频处理"，而是"低频抽帧推理 + 秒级统计"，推理成本与播放帧率解耦。

2. **两种热力图方案**：
   - **方案 E（Canvas叠加）**：前端播放原始视频 + WebSocket 推送结果 + Canvas 叠加，延迟低但前端复杂
   - **方案 F（服务端渲染）**：服务端叠加热力图后推流，前端极简但延迟 1-5 秒

3. **多路并发关键点**：每路只保留最新帧（忙就丢旧帧），才能稳定秒级输出。

4. **协议降级策略**：WebRTC（最低延迟） -> HTTP-FLV（低延迟） -> HLS（兼容性最好）。

5. **ZLMediaKit 的角色**：
   - 协议转换（RTSP -> WebRTC/FLV/HLS）
   - 抽帧服务（getSnap API）
   - SFU（多人观看）
   - 接收渲染后流（方案 F）
