## 难点 4 总结（RTSP 流稳定性处理）

### 1) 背景/目标

**背景：** 摄像头（RTSP/IP Camera/NVR）在公网/复杂网络下会出现抖动、断连、半断连（连接在但不出帧）、鉴权失败、NVR 切流等情况。
**目标：**

- 断连/卡住时能**自动恢复**（重连）
- 重连长期失败时能**提示用户**并**停止空耗资源**
- 下游推理仍坚持“**只要最新帧**、忙就丢旧帧”，避免排队导致越来越旧

------

### 2) 整体数据链路（含状态/告警流）

```
RTSP Camera/NVR
   │   (断连/抖动/无帧/鉴权失败/超时)
   ▼
[采集隔离单元 Worker: 线程/进程/FFmpeg子进程]
   │  产出：latest_frame + ts + seq + health
   ▼
[最新帧缓存（每路一个）]  —— 推理每秒取 1–3 次（只读 latest）
   │
   ├──> [推理 Worker] -> 检测结果（带 seq/ts） -> WebSocket/Redis -> 前端叠加
   │
   └──> [观看分发链路(可独立)] -> Web 大盘观看（原视频/转发流）
   
同时：
[Worker 健康上报/心跳/错误码] -> [Supervisor 看门狗/状态机/退避] 
   -> 状态变更事件（Online/Reconnecting/Offline/Cooldown + reason + next_retry_at）
   -> 前端提示 + 运维告警
```

------

## 你的理解变化（每点一句话）

- 你从“断了就重连”进化到“**重连失败要对用户可见**，不能黑屏装死”。
- 你意识到“**一直重连会空耗资源**”，需要阈值与 cooldown 来止损。
- 你把“错误处理”从代码里的 try/except，升级成“**状态机 + 退避 + 告警语义**”。
- 你开始区分“**可恢复错误 vs 不可恢复错误**”，避免对鉴权失败这类错误做无意义重试。

## 沉淀核心结论（可复用的 3–5 条）

1. **采集必须隔离且可杀**：只要存在“read 可能卡死”，就必须用进程/子进程 + 看门狗。
2. **重连必须有退避与上限**：否则网络差时会变成重连风暴，反而拖垮系统与摄像头。
3. **失败必须对用户可见且可操作**：Offline/Cooldown/原因/下次重试时间，配“立即重试/编辑/禁用”。
4. **错误要分层分类**：鉴权/URL 错这类“不可恢复”要立即停；超时/断网这类“可恢复”才重试。
5. **实时语义优先：只保最新帧**：采集再抖也不要让下游排队，否则“看起来在线但实际越来越旧”。
