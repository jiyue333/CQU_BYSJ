## 背景与目的

我们需要一个**对用户透明的媒体网关**，把 **RTSP 摄像头 + 视频文件** 统一成浏览器可播放的流（WebRTC/HLS 等），同时让**推理服务以 1–3fps 拉取 snapshot（JPEG bytes）**做检测；尽量避免“推理再拉整路视频流”或额外维护复杂旁路。

------

## 网关对比（面向本项目关键点）

| 网关           | 优点（贴合点）                                               | 缺点/代价                                                    | 适配度（本期） |
| -------------- | ------------------------------------------------------------ | ------------------------------------------------------------ | -------------- |
| **ZLMediaKit** | 通常具备**现成的截图/快照接口** + **RTSP 代理拉流**能力，容易直接落到“推理端按 stream_id 调 snapshot”这条路线；多协议互转覆盖面广 | 功能多、配置项多，运维/参数需要规范                          | ⭐⭐⭐⭐⭐          |
| **MediaMTX**   | 部署轻、定位清晰（媒体路由器）；有 on-demand/hook 机制，按需启停很方便 | **没有天然等价的 getSnap HTTP API 体验**，常见做法是你自己补一层“快照 sidecar”（脚本/小服务） | ⭐⭐⭐⭐           |
| **SRS**        | 直播/RTC 平台能力强，生态成熟（WebRTC/HLS/RTMP/HTTP-FLV/SRT 等） | RTSP 接入与“快照抽帧”路径往往更平台化，通常仍需要额外旁路（如 ingest/FFmpeg/自研抓帧）才能变成你要的 `getSnap` 模式 | ⭐⭐⭐            |

------

## 结论

**本期（当前架构：推理服务调 snapshot、网关对用户透明、尽量少旁路）选择 ZLMediaKit。**若未来要上更完整的直播/RTC 平台能力，再考虑 **SRS** 更合适。
